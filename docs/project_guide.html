The Big Picture — What Did You Build?

▼

You built a RESTful Web API that sits on top of the Hotel Booking Demand dataset from Kaggle. The dataset contains 119,390 real hotel booking records from two Portuguese hotels (a city hotel and a resort hotel), covering 2015–2017. Each record has 32 features — lead time, country, market segment, deposit type, cancellation status, etc.

What the API does (in plain English)

Imagine you're a hotel manager. This API lets you:

  1. Browse bookings — search, filter by hotel/country/month, paginate through 119K records
  2. Log incidents — record operational issues (complaints, overbookings, no-shows) against bookings. Full Create, Read, Update, Delete.
  3. Attach notes — managers write internal notes on any booking. Another CRUD resource.
  4. View analytics — cancellation rates, monthly demand trends, market segment performance. These run SQL aggregations behind the scenes.
  5. Get risk insights — a rule-based algorithm scores each booking's cancellation risk (0–1). You can fetch the highest-risk bookings.
  6. AI risk assessments — send a booking to OpenAI's GPT, which writes a natural-language analysis with recommendations. This is the "creative GenAI" feature.
Exam tip: When they ask "what does your API do?", lead with the business problem (hotel cancellation management), not the technology. Then layer in: "it's RESTful, uses FastAPI, has dual auth, AI integration, and a live dashboard."

The Dataset

Antonio et al. (2019) published this dataset. Key stats: 119,390 records, 32 columns, two hotels (Resort Hotel + City Hotel), date range 2015–2017. The is_canceled column is binary (0/1) — about 37% of bookings were cancelled. This high cancellation rate is what makes the risk-scoring feature meaningful.

System Architecture — How It All Fits Together

▼

Layered Architecture

The project follows a layered architecture pattern. Each layer has a single responsibility, and dependencies only flow downward. This means you can test each layer independently.

┌─────────────────────────────────────────────────────┐ │ CLIENT (browser, curl, Swagger UI, dashboard) │ └──────────────────────┬──────────────────────────────┘ │ HTTP Request ▼ ┌─────────────────────────────────────────────────────┐ │ MIDDLEWARE LAYER │ │ ├── CORS (allow cross-origin requests) │ │ ├── Rate Limiter (60 req/min via slowapi) │ │ └── Request Logger (method, path, status, latency) │ └──────────────────────┬──────────────────────────────┘ ▼ ┌─────────────────────────────────────────────────────┐ │ ROUTER LAYER (app/routers/) │ │ ├── bookings.py — GET (browse/filter) │ │ ├── incidents.py — full CRUD │ │ ├── notes.py — CRUD │ │ ├── analytics.py — aggregation queries │ │ ├── insights.py — risk scoring endpoints │ │ ├── ai_insights.py — LLM-powered analysis │ │ └── auth_router.py — token issuance │ └──────────────────────┬──────────────────────────────┘ ▼ ┌─────────────────────────────────────────────────────┐ │ AUTH LAYER (app/auth.py) │ │ ├── API Key check (X-API-Key header) │ │ ├── JWT Bearer check (Authorization header) │ │ └── FastAPI Depends() injection │ └──────────────────────┬──────────────────────────────┘ ▼ ┌─────────────────────────────────────────────────────┐ │ SCHEMA LAYER (app/schemas.py) │ │ Pydantic models validate input + serialise output │ └──────────────────────┬──────────────────────────────┘ ▼ ┌─────────────────────────────────────────────────────┐ │ SERVICE LAYER (app/services/) │ │ ├── risk_scoring.py — heuristic algorithm │ │ └── llm_analysis.py — OpenAI API integration │ └──────────────────────┬──────────────────────────────┘ ▼ ┌─────────────────────────────────────────────────────┐ │ ORM LAYER (app/models.py + app/database.py) │ │ SQLAlchemy models → SQL queries → SQLite │ └─────────────────────────────────────────────────────┘

Key architectural pattern: Dependency Injection

FastAPI's Depends() is the backbone. Every route that needs database access gets a fresh session via Depends(get_db). Every route that needs auth gets it via Depends(require_api_key). This is how you get:

  • Per-request DB sessions — each request gets its own session, no shared state, proper cleanup
  • Test isolation — tests override get_db to use an in-memory SQLite database
  • Separation of concerns — routes don't know how auth works, they just declare they need it
# How Depends() works — from database.py
def get_db():
    """Yields a DB session, closes it after the request."""
    db = SessionLocal()
    try:
        yield db        # Route gets this session
    finally:
        db.close()      # Always cleaned up, even on errors

# Used in routes like this:
@router.get("/bookings")
def list_bookings(db: Session = Depends(get_db)):
    return db.query(Booking).all()

database.py — The connection

import os
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker, DeclarativeBase

BASE_DIR = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
DEFAULT_DB_PATH = os.path.join(BASE_DIR, "hotel_bookings.db")
DATABASE_URL = os.environ.get("DATABASE_URL", f"sqlite:///{DEFAULT_DB_PATH}")

engine = create_engine(
    DATABASE_URL,
    connect_args={"check_same_thread": False},  # Needed for SQLite + async workers
    echo=False,
)

SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)

class Base(DeclarativeBase):
    pass

def get_db():
    """Dependency injection — yields a session, always closes it."""
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
check_same_thread=False — SQLite normally only allows the thread that created it to use it. FastAPI uses multiple async workers, so we disable this check. It's safe because SQLAlchemy's session management ensures proper access.

models.py — ORM models (key parts)

class Booking(Base):
    __tablename__ = "bookings"
    booking_id = Column(Integer, primary_key=True, autoincrement=True)
    hotel = Column(String, nullable=False, index=True)
    is_canceled = Column(Integer, nullable=False, default=0)
    lead_time = Column(Integer)
    # ... 32 columns mirroring the CSV ...

    # Relationships — one booking can have many incidents and notes
    incidents = relationship("Incident", back_populates="booking", cascade="all, delete-orphan")
    notes = relationship("ManagerNote", back_populates="booking", cascade="all, delete-orphan")

class Incident(Base):
    __tablename__ = "incidents"
    incident_id = Column(Integer, primary_key=True, autoincrement=True)
    booking_id = Column(Integer, ForeignKey("bookings.booking_id"), nullable=False, index=True)
    incident_type = Column(String, nullable=False)
    severity = Column(Integer, nullable=False)
    # ...
    # Database-level constraints:
    __table_args__ = (
        CheckConstraint("severity >= 1 AND severity <= 5", name="check_severity_range"),
        CheckConstraint("status IN ('open', 'investigating', 'resolved')", name="check_status_values"),
    )
Why denormalised? The bookings table keeps all 32 columns from the CSV in a flat structure rather than normalising hotel names, countries, etc. into separate lookup tables. At 119K records, the storage overhead is negligible, and it avoids complex JOINs for every query. The ORM makes it trivial to switch if needed.
Why cascade="all, delete-orphan"? If a booking is deleted, all its related incidents and notes are automatically deleted too. No orphaned records.

Quick check: Why does the Incident model have CheckConstraints?

CheckConstraints enforce rules at the database level. Even if someone bypasses the API and writes directly to the DB, severity must be 1–5 and status must be one of the allowed values. This is defence in depth — Pydantic validates at the API layer, CheckConstraints validate at the DB layer.

Authentication & Security

▼

Dual authentication strategy

The API supports two authentication methods, and any write endpoint accepts either one:

MethodHowBest for
API Key Send X-API-Key: hotel-booking-dev-key-2025 header Simple, stateless. Ideal for machine-to-machine or quick testing.
JWT Bearer POST your API key to /auth/token, get back a JWT. Send Authorization: Bearer <token> Session-based. Token expires in 30 minutes. More secure for sustained access.

The auth flow

Option A: Direct API Key Client → sends X-API-Key header → require_api_key() checks it → ✅ Access Option B: JWT Flow 1. Client → POST /auth/token { "api_key": "hotel-booking-dev-key-2025" } 2. Server → validates key → creates JWT with exp, iss, sub claims → returns token 3. Client → sends Authorization: Bearer <token> → require_api_key() checks it → ✅ Access

The core auth code (auth.py)

# Configuration
API_KEY = os.environ.get("API_KEY", "hotel-booking-dev-key-2025")
JWT_SECRET = hashlib.sha256(API_KEY.encode()).hexdigest()  # Derived from API key!

async def require_api_key(
    api_key: Optional[str] = Security(api_key_header),
    bearer: Optional[HTTPAuthorizationCredentials] = Security(bearer_scheme),
) -> str:
    # Try API key first
    if api_key is not None:
        if hmac.compare_digest(api_key, API_KEY):  # ← CONSTANT-TIME comparison
            return "api_key_user"
        raise HTTPException(status_code=403, detail="Invalid API key.")

    # Try Bearer token
    if bearer is not None:
        payload = verify_jwt_token(bearer.credentials)
        return payload.get("sub", "jwt_user")

    # Neither provided
    raise HTTPException(status_code=401, detail="Authentication required...")
Critical concept: hmac.compare_digest()
Normal string comparison (==) is vulnerable to timing attacks. It returns False as soon as it finds the first different character — so comparing "aaaa" vs "baaa" takes less time than "aaaa" vs "aaab". An attacker can guess the key one character at a time by measuring response times.

hmac.compare_digest() always takes the same amount of time regardless of where the strings differ. This is called constant-time comparison. It's an OWASP best practice.

JWT token creation

def create_jwt_token(subject: str = "api_user") -> dict:
    now = datetime.now(timezone.utc)
    expires = now + timedelta(minutes=JWT_EXPIRY_MINUTES)  # 30 min default

    payload = {
        "sub": subject,     # Who the token is for
        "iss": JWT_ISSUER,  # "hotel-booking-intelligence-api"
        "iat": int(now.timestamp()),      # Issued at
        "exp": int(expires.timestamp()),  # Expires at
    }

    token = pyjwt.encode(payload, JWT_SECRET, algorithm="HS256")
    return {"access_token": token, ...}
JWT secret is derived from the API key via SHA-256: JWT_SECRET = hashlib.sha256(API_KEY.encode()).hexdigest(). This means if you change the API key, all existing JWT tokens are automatically invalidated. Neat security property.

What's protected vs public?

Exam tip: "Why are GETs public?" — The API holds read-only public data from a published dataset. Keeping GETs public makes it easy to explore and demo. Write operations (creating incidents, deleting records) require auth because they modify data. This is a proportionate security model.

Quick check: What HTTP status code does the API return for a missing auth header vs an invalid API key?

401 Unauthorized means "you didn't identify yourself at all." 403 Forbidden means "you identified yourself but your credentials are wrong." This follows HTTP semantics correctly. The code: missing → 401 (with WWW-Authenticate header), wrong key → 403.

Risk Scoring — The Heuristic Algorithm

▼

How it works

The risk scoring service (risk_scoring.py) calculates a cancellation risk score from 0.0 to 1.0 for any booking. It uses 6 weighted factors derived from domain analysis of the dataset.

FactorConditionWeightRationale
Lead time> 300 days+0.25Very far-out bookings cancel more
Lead time> 200 days+0.15High but not extreme
Prev cancellations≥ 3+0.30Strongest predictor — pattern behaviour
Prev cancellations≥ 1+0.20Any history is a signal
Deposit type"No Deposit"+0.15No financial commitment
Market segment"Online TA"+0.10OTAs have higher cancel rates
Repeat guestFirst-time (0)+0.10No loyalty attachment
Booking changesZero changes+0.10May indicate less engagement

Scores are capped at 1.0 and classified into bands:

  • Low (0.0–0.29): Minimal concern
  • Medium (0.30–0.59): Monitor, consider outreach
  • High (0.60–1.0): Proactive intervention recommended

The full code

def calculate_risk_score(lead_time, previous_cancellations, deposit_type,
                         market_segment, is_repeated_guest, booking_changes):
    score = 0.0
    factors = []

    # Factor 1: Lead time
    if lead_time > 300:
        score += 0.25; factors.append("very_high_lead_time")
    elif lead_time > 200:
        score += 0.15; factors.append("high_lead_time")

    # Factor 2: Previous cancellations — strongest single predictor
    if previous_cancellations >= 3:
        score += 0.30; factors.append("multiple_previous_cancellations")
    elif previous_cancellations >= 1:
        score += 0.20; factors.append("previous_cancellation_history")

    # Factor 3: No deposit = lower commitment
    if deposit_type == "No Deposit":
        score += 0.15; factors.append("no_deposit")

    # Factor 4: Online TAs have higher cancellation rates
    if market_segment == "Online TA":
        score += 0.10; factors.append("online_ta_segment")

    # Factor 5: First-time guests cancel more
    if is_repeated_guest == 0:
        score += 0.10; factors.append("first_time_guest")

    # Factor 6: Zero changes = less engagement
    if booking_changes == 0:
        score += 0.10; factors.append("no_booking_changes")

    score = min(score, 1.0)  # Cap at 1.0

    if score >= 0.60: band = "high"
    elif score >= 0.30: band = "medium"
    else: band = "low"

    return round(score, 2), band, factors
Why a heuristic instead of machine learning? A rule-based heuristic is transparent and interpretable — you can explain exactly why a booking scored 0.65 ("it had very high lead time + no deposit + first-time guest"). An ML model is a black box. For a coursework viva, being able to explain the algorithm is more valuable than raw accuracy. The code also acknowledges this as a limitation and suggests ML as future work.

Quick check: What's the maximum possible risk score for a single booking?

The theoretical maximum is 0.25 + 0.30 + 0.15 + 0.10 + 0.10 + 0.10 = 1.00. But if all factors fired, the sum could go up to 1.00 exactly. The min(score, 1.0) cap ensures it never exceeds 1.0. Without the cap, a booking with >300 lead time AND 3+ cancellations AND all other factors would total 1.00 naturally.

AI Integration — GPT-Powered Risk Assessments

▼

The pipeline

When a client calls POST /insights/ai-risk-assessment/{booking_id}, here's what happens:

1. Fetch booking from database (by ID) ↓ 2. Calculate risk score using the heuristic (risk_scoring.py) ↓ 3. Build prompt with booking data + risk score + factors ↓ 4. Call OpenAI Chat Completions API (gpt-3.5-turbo) ↓ 5. Return structured response: score + band + factors + AI narrative If OpenAI fails → graceful degradation → template-based response

The router code (ai_insights.py)

@router.post("/ai-risk-assessment/{booking_id}", response_model=AIRiskAssessment,
              dependencies=[Depends(require_api_key)])
async def ai_risk_assessment(booking_id: int, db: Session = Depends(get_db)):
    # Step 1: Fetch the booking
    booking = db.query(Booking).filter(Booking.booking_id == booking_id).first()
    if not booking:
        raise HTTPException(status_code=404, detail=f"Booking {booking_id} not found")

    # Step 2: Calculate risk score (heuristic)
    score, band, factors = calculate_risk_score(
        lead_time=booking.lead_time or 0,
        previous_cancellations=booking.previous_cancellations or 0,
        deposit_type=booking.deposit_type or "",
        market_segment=booking.market_segment or "",
        is_repeated_guest=booking.is_repeated_guest or 0,
        booking_changes=booking.booking_changes or 0,
    )

    # Step 3: Build booking context dict for the LLM
    booking_data = {
        "hotel": booking.hotel, "lead_time": booking.lead_time,
        "arrival_date_month": booking.arrival_date_month,
        "market_segment": booking.market_segment,
        "deposit_type": booking.deposit_type,
        "adr": booking.adr or 0,
        # ... more fields ...
    }

    # Step 4: Call OpenAI (or fallback)
    llm_result = await generate_risk_assessment(booking_data, score, band, factors)

    # Step 5: Return structured response
    return AIRiskAssessment(
        booking_id=booking_id, hotel=booking.hotel,
        risk_score=score, risk_band=band, risk_factors=factors,
        ai_assessment=llm_result["assessment"],
        ai_model=llm_result["model"], ai_source=llm_result["source"],
    )

The LLM service (llm_analysis.py)

Graceful degradation (_template_fallback)
This is one of the most important design choices. If OpenAI is down, if the API key is wrong, if the network fails — the endpoint still returns a valid, schema-identical response. The fallback generates a template-based assessment using the same risk factors. The client never sees an error; the ai_source field changes from "openai" to "template" so you know which path was taken.
Why temperature: 0.3? — Temperature controls randomness. 0 = fully deterministic, 1 = creative. We want factual, consistent risk assessments, not creative writing. 0.3 gives slightly varied phrasing while staying factual.
Why httpx.AsyncClient not requests? — requests is synchronous; it would block the entire FastAPI event loop while waiting for OpenAI (up to 30 seconds). httpx is async-native, so other requests can be served while waiting for the LLM. This is non-blocking I/O.
Exam tip: The AI integration is at three levels: (1) ChatGPT was used as a development tool to help scaffold code, (2) Prism was used to draft the LaTeX report, and (3) GPT is embedded as a runtime feature of the API itself. Level 3 is the "creative application of GenAI" the rubric rewards. You built an AI-powered product, not just used AI to build a product.

Quick check: What happens if someone calls /insights/ai-risk-assessment/1 but no OPENAI_API_KEY is configured?

The endpoint always succeeds. The _template_fallback() function generates a structured assessment using the same heuristic data. The response has ai_source: "template" and ai_model: "template-based (no LLM API key configured)". This is graceful degradation — the API contract never breaks.

What it is

A single-page HTML file (static/index.html) served by FastAPI at /dashboard. It's vanilla HTML/CSS/JavaScript — no frameworks, no build step. Uses Chart.js for visualisations.

# Serve the dashboard at /dashboard
@app.get("/dashboard", include_in_schema=False)

    return FileResponse(os.path.join(static_dir, "index.html"))

include_in_schema=False hides this from Swagger docs — it's a UI page, not an API endpoint.

  2. Monthly Demand — Chart.js bar chart from /analytics/monthly-demand

Why include a dashboard? It demonstrates full-stack delivery — the API isn't just an abstract JSON producer, it powers a real visual interface. It shows the API's data flowing into something a hotel manager could actually use. The AI Analyse button ties the dashboard directly to the GPT integration.

Test isolation — how it works

# conftest.py — override the database for tests
@pytest.fixture
def db_session():
    """Create a fresh in-memory SQLite database for each test."""
    engine = create_engine("sqlite:///:memory:")  # ← In-memory, not on disk!
    Base.metadata.create_all(bind=engine)
    Session = sessionmaker(bind=engine)
    session = Session()
    yield session
    session.close()

@pytest.fixture
def client(db_session):
    """Test client with the overridden DB."""
    def override_get_db():
        yield db_session
    app.dependency_overrides[get_db] = override_get_db  # ← The magic
    yield TestClient(app)
    app.dependency_overrides.clear()
Key pattern: dependency_overrides
FastAPI lets you replace any Depends() dependency for testing. We replace get_db with a version that returns an in-memory SQLite session. Each test gets a fresh, empty database. Tests run in 0.87 seconds total because there's no disk I/O.

What the tests prove

  • Auth gates work — POST/PUT/PATCH/DELETE without a key → 401
  • Validation works — invalid severity (6) → 422, missing fields → 422
  • FK integrity — creating an incident for a non-existent booking → 404
  • Filters work — filter by hotel, by status, by month — correct results returned
  • AI fallback works — the template response has the correct schema when no OpenAI key
  • Edge cases — empty results, zero records, boundary values
Acknowledged limitation: No end-to-end (E2E) tests. The test pyramid has unit/integration tests but no browser-based or multi-step workflow tests. This is mentioned in the report as future work.

Deployment & DevOps

▼

Railway — how it works

Railway is a Platform-as-a-Service (PaaS). The deployment flow:

  1. You push to main on GitHub
  2. Railway detects the push (webhook)
  3. Nixpacks auto-detects it's a Python project (from requirements.txt)
  4. Builds and deploys automatically
  5. HTTPS is automatic via Railway's reverse proxy
  6. Environment variables (API_KEY, OPENAI_API_KEY, JWT_SECRET, etc.) are injected at runtime

Twelve-Factor App alignment

The project follows several principles from the Twelve-Factor App methodology (Wiggins, 2011):

FactorHow we implement it
I. CodebaseOne repo on GitHub, tracked in version control
II. Dependenciesrequirements.txt declares all dependencies explicitly
III. ConfigAll config via environment variables (API_KEY, DATABASE_URL, OPENAI_API_KEY, etc.)
IV. Backing servicesSQLite is an attached resource; swap to PostgreSQL by changing DATABASE_URL
V. Build, release, runRailway handles build → release → run stages

Health check endpoint

@app.get("/health")
def health_check():
    try:
        db = SessionLocal()
        db.execute(sqlalchemy.text("SELECT 1"))  # Quick DB probe
        db.close()
        db_status = "connected"
    except Exception as exc:
        db_status = f"error: {exc}"
    return {"status": "healthy", "database": db_status, "version": "1.0.0"}
Why a health check? Railway (and most PaaS platforms) can ping this endpoint to confirm the service is alive. If it returns an error, the platform can auto-restart. It also verifies database connectivity, not just that the Python process is running.

Viva Preparation — Likely Questions & Answers

▼

"What does your API do?"

"It's a RESTful intelligence layer over 119,000 real hotel booking records. Hotel managers can browse and filter bookings, log operational incidents with full CRUD, view analytics like cancellation rates and demand trends, and get AI-powered risk assessments that predict which bookings are likely to cancel — with actionable recommendations. It's built with FastAPI, uses dual authentication, and is deployed live on Railway."

"Why did you choose FastAPI?"

"Three reasons: it auto-generates interactive Swagger documentation from Python type hints — the examiner can explore every endpoint without reading docs. It has built-in request validation through Pydantic, so invalid data is rejected before it hits the database. And it supports async natively, which I needed for the non-blocking OpenAI API calls."

"Explain your authentication system."

"Dual auth — API keys for quick access, JWT tokens for session-based access. API keys are checked using constant-time comparison with hmac.compare_digest() to prevent timing attacks. JWT tokens follow RFC 7519 with 30-minute expiry. The JWT secret is derived from the API key via SHA-256, so changing the key invalidates all tokens. Read endpoints are public; write endpoints require auth."

"How does your AI integration work?"

"There's a four-step pipeline. First, fetch the booking from the database. Second, calculate a heuristic risk score using six weighted factors. Third, build a structured prompt with the booking data and risk score, and send it to OpenAI's Chat Completions API. Fourth, return the combined structured data plus the LLM narrative. If OpenAI is unavailable, the endpoint gracefully degrades to a template-based response with the same schema — the API contract never breaks."

"What are the limitations of your project?"

"SQLite limits write concurrency — in production I'd use PostgreSQL. There's no OAuth2 scopes or RBAC — all authenticated users have the same permissions. The risk heuristic is rule-based, not ML-trained — a production system would use a trained classifier. The LLM calls are synchronous within a request — an async task queue like Celery would be better for long-running AI analysis. And there's no end-to-end testing."

"What would you do differently / as future work?"

"Migrate to PostgreSQL for production concurrency. Add OAuth2 scopes for fine-grained permissions. Train an ML model on the cancellation data instead of a heuristic. Use an async task queue for LLM calls so the client doesn't wait. Add WebSocket notifications for real-time risk alerts. And implement comprehensive E2E testing."

"How did you use Generative AI in this project?"

"Three levels. First, ChatGPT (GPT-4) as a development tool — it helped scaffold the project structure, debug issues, and iterate on the architecture. Second, Prism for drafting the LaTeX report — I provided detailed technical context and it generated the document, which I then reviewed and refined. Third, and most importantly, GPT-3.5-turbo is embedded as a runtime feature of the API itself — the AI risk assessment endpoint sends booking data to the LLM and returns natural language analysis. That third level is the creative application — the API is AI-powered, not just AI-built."

"What's the difference between /insights/high-risk-bookings and /insights/ai-risk-assessment?"

"High-risk-bookings is a GET endpoint that scans all non-cancelled bookings through the heuristic algorithm and returns those above a risk threshold. It's pure computation — fast but gives just numbers and factor labels. AI-risk-assessment is a POST endpoint for a single booking that combines the heuristic score with a GPT narrative — it explains WHY the booking is risky in plain English and gives actionable recommendations. One is a batch scanner, the other is a deep-dive analyser."

"Why is POST used for AI risk assessment instead of GET?"

"The AI risk assessment has a side effect — it calls the OpenAI API, which costs money and changes state externally (token usage, billing). GET should be safe and idempotent, meaning it shouldn't cause side effects. POST is appropriate because each call triggers an external API call with a cost. It also allows us to require authentication, which aligns with this being a premium feature."
